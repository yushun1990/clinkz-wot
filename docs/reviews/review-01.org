#+title: Architecture Review 01: Core, Runtime, and Binding Boundaries
#+date: 2026-07-16
#+status: Complete
#+reviewed_design: v4.7 candidate

* Executive verdict

The v4.7 candidate is not yet implementation-ready for WP-200, WP-300, WP-400,
or WP-600. It has the correct general direction, especially compiled binding
ownership and =ProducerEmission=, but it still leaves implementers to choose the
host subscription driver, emission scheduling owner, collection subscription
fallback, and global-versus-binding emission state. It also freezes the
ambiguous =BindingRequest= name while normative prose still lists
=EventBroker= as a core and Servient component.

Foundation and handler implementation must remain paused until these decisions
are incorporated into one coherent design revision and the affected gates are
re-evaluated. The pause prevents new WP-100 code from depending on a binding or
subscription boundary that will immediately change.

The issues recorded in =docs/thinking/2026-07.org= are confirmed. They expose
architecture defects rather than optional optimizations. The accepted decisions
are indexed in =docs/ADR/core.org=.

* Scope and method

This review compared:

- the active v4.7 design, API ownership matrix, state machines, limits,
  performance manifests, and work-package DAG;
- the current core, Servient, shared compiler, Discovery, and Zenoh
  implementations; and
- the concerns recorded on 2026-07-14 and 2026-07-15 under =docs/thinking=.

The review is an architecture and implementation-conformance audit. It does not
claim benchmark results for code that has not yet been migrated.

* What v4.7 already gets right

- =FORM-OWNER-001= assigns each compiled inbound plan to exactly one
  =BindingId=.
- =PRODUCER-EMIT-001= replaces opaque sink fan-out with typed emission values,
  bounded per-binding results, shared payload ownership, and retained cursors.
- WP-200 separates plan compilation from binding execution, and forbids
  per-interaction TD scans in the target.
- WP-300 introduces one binding SPI and prevents new callers from entering the
  legacy publication bridge.
- WP-400 owns Servient lifecycle and handler activation; WP-600 owns concrete
  Zenoh migration.
- The target Discovery contract is client-only and correctly defers Directory
  service topology and storage.

These decisions should be retained. The new ADRs close the ownership gaps around
them rather than replacing the entire v4.7 direction.

* Resolution of the recorded concerns

| Concern | Review result | Accepted resolution |
|-
| =EventBroker= naming | Confirmed. It is neither a durable broker nor only an event dispatcher. | Remove it. Servient owns an =EmissionCoordinator= and replaceable =EmissionDispatchPolicy= because the path also carries property changes. |
| Thing lookup and serial sink calls | Confirmed, with additional correctness defects. | Use the handle's generation-safe record and compiled publication target set; isolate binding lanes and add slow-binding workloads. |
| Concrete dispatcher in core | Rejected by the target boundary. | Core owns emission values and per-binding SPI only; Servient owns host and constrained coordination strategies. |
| =PublisherSink= provenance | Currently derived by each Zenoh server binding rescanning TD forms during =serve=. | Derive targets from finalized forms, exact plan owner, binding registration, and active route generation. Remote subscribers remain binding-internal. |
| Core-owned subscription queue | Confirmed as the wrong abstraction. | Binding supplies a pull-capable driver or portable slot; Servient owns the public facade and accounting; optional queue adapters stay outside core. |
| =BindingRequest= naming | Confirmed as directionally ambiguous. | Rename the target envelope to =OutboundRequest= and remove the old public name in WP-300. |
| Eager =subscribe_all_events= fan-out | Confirmed and contrary to the active root-form rule. | Standard collection methods execute one selected Thing-level plan. Missing root forms fail explicitly; N-way fallback is deferred until separately designed. |

* Design-closure blockers

** AR-D01: Emission ownership is contradictory (critical)

=docs/design.md= still names =EventBroker= in the interaction core and Servient
composition, while =docs/api-ownership.csv= marks =PublisherSink= for removal
and does not give =EventBroker= or =EventStream= a target disposition. This
leaves the concrete coordinator owner undefined.

Required action: apply ADR-0002, remove normative broker references, add exact
ownership rows for every removed and replacement item, and make Servient the
only global emission coordinator owner.

** AR-D02: Global and binding-local emission state is conflated (critical)

The v4.7 skeleton passes =ServerEmissionSlot= to one =PollServerBinding=, but
the same slot is described as owning the local-subscriber cursor and the cursor
over all binding targets. One concrete binding cannot own Servient-local
subscribers or other bindings' progress.

Required action: split a private Servient =EmissionRecord= from core's public
=BindingEmissionSlot= and specify separate state machines and cleanup ownership.

** AR-D03: The host subscription receive SPI is not frozen (critical)

=HostSubscriptionStart= is named but has no exact receive object, polling
contract, stop contract, field schema, or ownership transfer. The public
=Subscription= method surface is also absent from the frozen API skeleton.
Implementers would have to invent a queue, channel, stream, or trait object.

Required action: apply ADR-0003 and freeze =HostSubscriptionDriver=,
=HostSubscriptionStart=, =SubscriptionItem=, the non-=Clone= Servient facade,
and the id reservation/install transaction before WP-300 begins.

** AR-D04: Subscription identity and storage ownership are incomplete (high)

The design says =SubscriptionId= is scoped to the consumed Thing but also places
it in metadata returned by the binding. It does not state who allocates it. The
storage text permits multiple physical strategies but does not assign transport
credit, ingress buffers, engine accounting, and public polling to exact owners.

Required action: Servient reserves the id and budgets before the binding call;
the binding echoes the identity and owns protocol-local flow control; core owns
limits and status semantics.

** AR-D05: Collection subscription planning is incomplete (critical)

The active design correctly treats Thing-level forms as first-class, but it does
not freeze whether the standard method may fan out, how native collection
capability is represented, or how a fallback reserves, rolls back, orders,
stops, and reports N subscriptions.

Required action: apply ADR-0004. The conforming standard method is native
root-plan only. Explicit fan-out is not part of the initial surface.

** AR-D06: Directional request naming is frozen incorrectly (medium)

WP-300 currently says to keep =BindingRequest=. This creates avoidable churn and
does not distinguish an engine-to-client-binding execution envelope from other
binding messages.

Required action: apply ADR-0005 and update every normative reference to
=OutboundRequest= before code migration.

** AR-D07: State and performance evidence is incomplete (critical)

There is no Producer emission coordinator/publication state machine. Current
performance workloads measure local subscriber fan-out but not compiled target
lookup, 1/4/16/64 binding publication scaling, one slow/full binding while
unrelated bindings progress, maximum-form exposure construction, or native
collection subscription behavior.

Required action: add the state machines, limits, workload identities, harness
cases, and evidence keys before the state, performance, and DAG gates close.

* Current implementation findings

These findings distinguish implementation debt from the design blockers above.
They remain relevant migration tests even when the target design already points
in the correct direction.

** AR-I01: Property and event routes can cross-publish (critical)

=core/src/event.rs:131= keys sinks by =ThingId= and =EventName= only.
=protocol-bindings/protocols/zenoh/src/server.rs:630= maps both property and
event names into that key, and =servient/src/handle.rs:526= routes property
change emission through the event method. A property and event with the same
name can receive each other's payloads.

Migration evidence must use full =AffordanceTarget=, =EmissionKind=, route
generation, and binding generation.

** AR-I02: Publication cleanup is not generation- or binding-safe (critical)

=EventBroker= has only whole-Thing and whole-name removal. Zenoh shutdown calls
whole-Thing removal, so one binding can delete another binding's targets. Late
callbacks for a destroyed and recreated textual Thing id cannot be isolated.

Migration evidence must prove tokenized removal and stale-generation rejection.

** AR-I03: Current publication has avoidable head-of-line blocking (high)

Registration copies the existing sink slice on every append
(=core/src/event.rs:197=), giving quadratic construction for a large target
set. Publication invokes every sink serially (=core/src/event.rs:247=), and the
Zenoh sink blocks in =session.put(...).wait()=
(=protocol-bindings/protocols/zenoh/src/server.rs:1165=). Failures collapse into
one generic error and lose per-binding outcomes.

** AR-I04: Core hard-wires subscription storage and violates constrained rules (critical)

=core/src/event.rs:303= combines =VecDeque=, =Arc=-backed =WotLock=,
drop-oldest, one waker, sender, stream, and merge behavior. The queue begins
without reserved storage, and =push= may allocate while the no-std lock is inside
a =critical_section::with= call. This violates =CONCUR-CRIT-001= and prevents
pull-native, credit-based, mailbox, fixed-ring, and rendezvous bindings.

** AR-I05: Subscription control does not control the wire resource (critical)

=Subscription::stop= only flips a local queue flag. Wire guards live separately
in =ConsumedThingHandle= and are keyed by =AffordanceTarget= rather than
=SubscriptionId=. A second subscription can replace the first guard, and
fallible cleanup has no retained =CleanupOutcome=.

** AR-I06: Local merge has correctness and fairness defects (critical)

=Subscription::poll_next= and =EventStream= always scan from index zero, so a
hot first source can starve later sources despite the round-robin documentation.
Merged =Subscription::is_empty= uses =all(has_buffered)=, which reports the
wrong result for common mixed and all-empty cases. Async merged polling rebuilds
a leaf vector on each poll, and cloned receivers compete for one waker slot.

These types are removed rather than promoted into the target architecture.

** AR-I07: Collection execution bypasses native protocol support (critical)

=servient/src/handle.rs:856= always opens one =SubscribeEvent= operation per TD
event. Zenoh already advertises =SubscribeAllEvents= and maps it to one native
subscriber, but the handle never selects that operation. =observe_all_properties=
is promised by the target surface but is absent from the current Consumer
handle.

** AR-I08: Consumer planning is repeated on the hot path (high)

Servient scans forms and clones one, core validates it and linearly scans
bindings, and Zenoh validates and compiles again. The Zenoh cache key does not
fully identify Thing/base and generation context and is not admitted by the
resource policy. This is both a boundary violation and a possible incorrect
relative-target cache hit.

Required migration: WP-200 compiles one immutable plan and =OutboundRequest=
contains only selected execution facts.

** AR-I09: Callbacks and lifecycle ordering violate the target (critical)

Current inbound dispatch invokes security providers and handlers while holding
Thing state locks. Exposure starts bindings before servable registry
publication, and destroy shuts bindings down before marking the Thing draining.
These create reentrancy deadlocks and request race windows.

Required migration: all callbacks occur outside engine guards; the frozen
prepare/readiness/activate/commit and drain-first state machines become the only
lifecycle path.

** AR-I10: Discovery still contains service behavior (high)

The Discovery crate exports =InMemoryDirectory=, storage helpers, and
=LocalDiscoverer=. =ServientBuilder= creates this service implicitly when no
client is configured. This contradicts the already-correct client-only target
and remains a WP-500 removal requirement.

** AR-I11: Feature and dependency boundaries are not implemented (high)

Servient has a normal optional dependency on the concrete Zenoh crate for test
integration, while its normal graph should remain protocol-neutral. Its
no-default public surface does not yet expose =StaticServient=. Current Zenoh
drop cleanup uses an unbounded global channel/worker rather than reserved
cleanup ownership.

Concrete integration tests should move to the concrete binding or umbrella
crate; constrained and cleanup fixtures must reject empty-placeholder and
unbounded implementations.

* Target module responsibility map

| Layer | Target modules | Primary responsibility |
|-
| Foundation | =resource=, =budget=, =time=, =generation= | Profile-independent accounting primitives only. |
| TD/TM | document model, builders, Serde, validation, pure defaults, URI values | Lossless W3C documents and validation; no planning or runtime. |
| Core handlers | =handler/context=, =handler/traits=, =handler/cancellation=, =handler/step=, =handler/storage= | Operation-specific semantic contracts and portable state. |
| Core binding SPI | =binding/client=, =binding/server=, =binding/registration=, =binding/subscription=, =binding/emission=, =binding/state= | Owned protocol-neutral I/O values, traits, slots, outcomes, and generations. |
| Core plans | =plan/logical=, =plan/candidate=, =plan/route= | Immutable plan values only. |
| Shared compiler | compiler, capability index, form/security/target resolution, URI templates | Build and select plans; never execute them. |
| Servient | producer, consumer, subscription, emission, dispatch, lifecycle, registry, =runtime/host=, =runtime/static= | Application facade, orchestration, scheduling policy, lifecycle, and cleanup ownership. |
| Discovery | client values, sessions, processes, watches, publisher client, adapters | Directory/Discovery client contract only. |
| Zenoh | compiler, client, server, subscription, emission, auth, lifecycle, =runtime/host=, =runtime/pico= | Zenoh-specific planning extensions and I/O. |
| Umbrella | feature and re-export modules | Public composition only. |

The intended runtime flows are:

#+begin_example
TD/TM -> shared compiler -> immutable core plan -> Servient coordinator
                                             -> selected binding generation

binding subscription driver <-> Servient Subscription facade <-> application
#+end_example

No arrow permits a binding to reparse the TD or a core value module to choose a
runtime queue.

* Gate impact

| Gate | Result | Reason |
|-
| GATE-1 API ownership | Blocked | Missing/remnant broker items, wrong Subscription owner, undefined host driver, and =BindingRequest= rename. |
| GATE-2 states and cleanup | Blocked | Missing emission coordinator/publication states and incomplete host subscription install/stop ownership. |
| GATE-3 Directory client boundary | Design decision retained | The target boundary is sound; current service/default implementation remains scheduled for WP-500 removal. The new revision still needs an unaffected-scope review record. |
| GATE-4 resource limits | Blocked for affected packages | Driver storage ownership, collection target bounds, coordinator/binding slots, and slow-lane capacity must be exact. |
| GATE-5 performance | Blocked | Missing target-lookup, binding-scaling, slow-binding isolation, exposure construction, and native collection workloads. |
| GATE-6 implementation DAG | Blocked | WP-200/300/400/600 still use the old request name and conflate emission/subscription owners. |

* Required migration order

1. Incorporate ADR-0001 through ADR-0005 into a new design revision, API
   ownership matrix, state machines, limits, performance manifests, and DAG.
2. Re-run structured design checks and an independent adversarial review; close
   only same-revision gates with executable evidence.
3. Complete the foundation additive refresh required by the new revision.
4. Complete WP-100 handler, codec, security, cancellation, and lock-isolation
   contracts without importing downstream runtime policy.
5. WP-200 compiles immutable plans, capability indexes, publication targets,
   and root collection subscription plans; eliminate repeated TD scans.
6. WP-300 implements =OutboundRequest=, binding registrations, host and poll
   subscription drivers, split emission slots, state machines, and bounded
   cleanup.
7. WP-400 implements Servient lifecycle, =Subscription=, emission coordination,
   host/static strategies, and removes handler-side push paths.
8. WP-500 removes Directory service behavior from the client crate. After
   WP-300, this package may proceed independently of WP-400 when the DAG records
   that independence.
9. WP-600 migrates Zenoh and zenoh-pico to compiled plans, native collection
   drivers, binding publication, and bounded cleanup; remove =PublisherSink=.
10. WP-700 removes every migration bridge, composes the umbrella API, and runs
    final conformance and performance evidence.

Do not patch =EventBroker= or the core queue into a second target architecture.
Only urgent correctness tests may be added to document the old behavior before
its dependency-ordered removal.

* Exit criteria for this review

The architecture-review action is complete when:

- all five ADRs are accepted and referenced by the active revision;
- every affected public item has one owner, path, feature cell, and migration
  disposition;
- the host subscription driver and both emission slot roles have exact schemas;
- collection methods have native-root-only semantics and no implicit fallback;
- emission and subscription state machines cover cancellation, stale
  generations, terminal retention, and cleanup transfer;
- resource and performance artifacts exercise the newly identified limits and
  bottlenecks; and
- the revised work-package DAG can be executed without implementers choosing a
  queue, scheduler, fallback, or request name locally.
