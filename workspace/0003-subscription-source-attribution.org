#+TITLE: Subscription Source Attribution
#+STATUS: MIGRATED

* Context

Subscription APIs may need to represent samples originating from multiple
affordances or concrete protocol-side sources while preserving a
protocol-neutral application-facing contract.

* Observation

A merged or multiplexed subscription needs a reliable way to identify which
logical source produced each delivered item.

The existing use of affordance-level identity may not be sufficient if one
logical affordance can map to multiple concrete runtime sources.

* Questions

- What protocol-neutral identity should accompany each subscription item?
- Is affordance identity sufficient?
- Must attribution distinguish forms, compiled routes, or runtime source
  instances?
- Which attribution belongs in the compiled plan rather than the runtime
  subscription API?
- Can protocol bindings multiplex efficiently without forcing the core runtime
  to model protocol-specific topics or resources?

* Discussion

Keep the application-facing contract protocol-neutral.

Do not expose MQTT topics, Zenoh key expressions, WebSocket channels, or other
protocol-specific source identifiers through the core API merely to support
attribution.

Prefer attribution derived from compiled protocol-neutral planning artifacts.

Further discussion is required before choosing the exact identity type and its
ownership boundary.

* Decision

The application-visible source identity is the exact protocol-neutral
=AffordanceTarget=. =CollectionSourceAttribution= is the planning capability
proof and standard collection operations accept only
=ExactAffordanceTarget=.

A compiled binding artifact may map multiple protocol-side sources to the same
logical target. Protocol topics, key expressions, channels, and route-instance
identities remain inside the binding artifact or bounded diagnostic status;
they do not enter =SubscriptionItem=. Observably different WoT semantics must
be modeled as different logical targets or by an explicitly designed
extension.

* Migration

Migrated into ADR-0003, ADR-0004, =docs/spec/planning.md=, and
=docs/spec/binding-spi.md=. The current eager per-affordance merge code remains
an implementation deviation assigned to later work packages, not an open
identity decision.
