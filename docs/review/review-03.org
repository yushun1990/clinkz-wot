#+title: Architecture Review 03: v4.9 Interim Refactor Admission
#+date: 2026-07-19
#+review_id: architecture-review-03-v4.9-interim
#+status: Blocking
#+reviewed_design: v4.9 architecture-closure candidate

* Executive verdict

The v4.9 architecture backbone is a material improvement over the rejected
v4.8 candidate, but the complete design is not yet implementation-ready. The
target module boundaries, compiled-plan lifecycle, Cargo-linked Protocol
Binding model, and engine-orchestrated route model are now coherent enough to
guide the remaining design work. They are not yet precise enough to let
independent implementers produce one compatible refactor without making new
public-API and ownership decisions.

GATE-1, GATE-2, and GATE-6 remain open. GATE-3 remains closed only as the
previously reviewed Directory-client carry-forward. GATE-4 and GATE-5 are not
concluded by this interim review because their resource and performance
projections were being completed in parallel.

Runtime migration must remain paused. Closing syntactic artifact checks is not
equivalent to closing the semantic contradictions below.

* Scope and evidence

This independent review compared the current versions of:

- =docs/thinking/2026-07.org= and =PLAN.md=;
- the complete =docs/architecture/= backbone;
- ADR-0001 through ADR-0011;
- =docs/spec/planning.md= and =docs/spec/binding-spi.md=;
- the still-active residual requirements in =docs/design.md=;
- API ownership, state-machine, requirement, gate, resource, performance, and
  work-package projections; and
- the migration packages under =docs/work-packages/=.

The review used adversarial traces for registration construction, expose
publication, route-call cancellation, subscription stop/drop, cleanup handoff,
constrained slot progress, and executor loss. It distinguishes a structurally
valid projection from a Rust contract that can actually preserve the claimed
owner.

At the end of the review, an independent run of
=./tools/check-design-artifacts.sh= did not complete successfully. It stopped in
the architecture ADR check because the checker still required
=fn start_stop_subscription(= while the active specification and ownership path
use =start_subscription_stop=. This is a projection/checker integration defect,
not evidence that any gate passes.

* Resolved architecture findings

The following v4.8 blockers now have a clear v4.9 architecture decision:

- Planning is a dedicated =clinkz-wot-planning= responsibility. Core owns
  protocol-neutral compiler/artifact SPI values, while planning owns shared
  algorithms and Servient owns mutable plan-set lifetime.
- A plan set has explicit build, freeze, publication, drain, pin, lazy-artifact,
  and reclamation ownership. Hot execution may not rescan a TD or all bindings.
- A Protocol Binding is an ordinary Cargo-linked crate installed through one
  complete startup registration. Dynamic Rust trait-object loading, hot unload,
  and =build.rs= discovery are outside v1.
- Producer routes are engine-orchestrated and route-scoped. Bindings do not
  receive application dispatch authority, and one route owns one accept cursor
  and waker lease.
- Host response and publication constructors now have an explicit typed
  pre-acceptance rejection direction, so an unaccepted owned input need not be
  lost.
- The state projection now names route lifecycle, readiness, active-route
  driving, binding-call cleanup, and cleanup-task ownership. These additions
  expose the remaining semantic conflicts instead of leaving them implicit.

No crate dependency inversion was found in the selected compiler boundary:
core defines the portable SPI, planning consumes it, Servient coordinates its
lifetime, and concrete bindings implement it.

* Remaining blockers

** AR3-01: Accepted decisions and the active binding specification conflict

ADR-0006 still freezes =BindingCallSettlement<T>= as exactly two variants,
=Returned(CoreResult<T>)= and =Cancelled { retry_class, cleanup }=. The active
=docs/spec/binding-spi.md= freezes four different variants:
=TerminalValue(T)=, =CleanupComplete(CleanupRecord)=,
=TransferRequired(CleanupTransferRequest)=, and
=ResidualExternalState(CleanupRecord)=. The specification also refers to a
=BindingCallSettlement::LateValue= branch that is not present in that exact
enum.

ADR-0003's accepted host subscription shape omits the =Context= and independent
=CleanupPhaseContext= now required by the active specification. Neither ADR is
marked amended or superseded for these exact surfaces. ADR-0007 explicitly
states that a conflict blocks implementation and is not resolved by file
precedence. An implementer must currently choose which accepted contract to
follow.

Required closure: amend ADR-0003 and ADR-0006 explicitly, retain one exact
settlement enum and late-value rule, and make every checker and API path use the
same subscription-stop spelling and signature.

** AR3-02: Complete registration ownership is named but not constructible

=docs/spec/binding-spi.md= lists the required contents of
=HostBindingRegistration= and =StaticBindingRegistration<B>=, but it does not
freeze the fields, component wrapper types, constructor parameters, validation
result, or downstream-construction bounds. The only exact implementation block
shown for both bundles is the collection-subscription capability accessor.
=docs/api-ownership.csv= freezes a =::new= path, but a path row contains no Rust
signature or bundle schema.

Consequently the required fake third-party crate cannot deterministically
construct the promised complete bundle from the documentation alone. Teams
would still decide how compiler, client/server execution, contributor,
compatibility identity, profile cells, footprints, ingress, reactor, status,
and cleanup policy are represented and erased.

Required closure: freeze complete host and static constructor/builder schemas,
component optionality and trait bounds, validation errors, accessors consumed by
planning and Servient, and the host-erased versus static artifact representation.

** AR3-03: The atomic activation gate has no executable public surface

The architecture, ADR-0010, planning specification, binding specification, and
state projection all require Servient to atomically publish the produced
registry entry, plan set, and every route's activation gate. The exact
=ServerBinding= trait has prepare, readiness, activate, commit, accept, abort,
shutdown, response, and publication methods, but no operation or capability by
which Servient opens a committed route's gate at the registry linearization
point. =RouteCommitOutcome::Serving(A)= is also named serving even though the
route must remain non-serving until the later atomic publication.

Residual =docs/design.md= instead says bindings observe final registry state
through =BindingContext=, while the architecture forbids bindings from owning
or observing the Servient registry and the exact route trait does not receive
such a context. The state transition therefore describes an atomic action that
the frozen SPI cannot perform.

Required closure: define one explicit gate token/capability and an infallible or
ownership-preserving release operation that participates in the Servient
commit, or revise the transaction model so the same boundary is implementable
without registry observation by a binding.

** AR3-04: Route-call cancellation can discard the guard needed by cleanup

=start_readiness=, =activate=, and =commit= move a prepared or active guard into
a generic =HostBindingCallBox<T>=. Their ordinary terminal outcomes return the
predecessor or successor guard, but cancellation returns
=BindingCallSettlement<T>=. The =CleanupComplete= and
=ResidualExternalState= branches contain only a status record; they cannot
return the guard held by the call.

ADR-0011 requires readiness cancellation and prepared-route abort, and likewise
call cancellation and stage-appropriate route cleanup, to be independent
obligations with independent reservations, phase contexts, causes, and
deadlines. A readiness call cannot use its one cancellation phase to silently
perform the later prepared-route abort. Therefore a synchronously completed
call cancellation can make the following abort/shutdown transaction impossible.
The state projection claims that the predecessor guard remains available, but
the exact type cannot carry it.

Required closure: use lifecycle-specific cancellation settlements that return
the retained guard or an explicit cleanup successor, or define another exact
owned extraction/continuation contract. Every branch must be traceable to
abort, shutdown, acknowledged transfer, or certified no-resource state.

** AR3-05: Subscription cleanup reports transfer before it can transfer work

=HostSubscriptionDriver::start_stop= and =poll_stop= return =CleanupOutcome=.
Its active exact residual definition in =docs/design.md= contains
=PendingCleanup(CleanupRecord)=, but no provisional =TransferRequired= branch.
The methods operate through =Pin<&mut Self>=, so returning the record does not
move the complete driver to an acknowledged cleanup owner. ADR-0011 explicitly
states that a record is not work and that pending is legal only after the
complete driver moves.

The same ambiguity affects constrained cleanup methods until their exact slot
transfer and acknowledgement signatures are supplied. A state-machine
composition cannot repair a return type that permits record-only pending.

Required closure: add a provisional driver/slot transfer result and exact
accept/reject handshake, or make stop retain ownership in its current record
until a separately specified atomic move completes.

** AR3-06: Constrained equivalence remains prose rather than an exact API

The constrained section of =docs/spec/binding-spi.md= explicitly calls its code
a semantic shape and leaves request, route lifecycle, response, emission,
cancellation, terminal acknowledgement, and =PollBindingCleanup= methods behind
comments. The ownership matrix supplies names and paths, but not receivers,
arguments, associated-type bounds, slot state transitions, consuming-start
rejection shapes, or return types.

This is especially important for =StaticBindingRegistration<B>= and
=BindingArtifact=: the documents do not freeze how one generic =B= supplies
compiler, optional client/server roles, contributor, heterogeneous artifact
state, and all caller-owned slots without host erasure. Host/constrained
semantic equivalence therefore cannot yet be compiled or tested from the
design.

Required closure: provide complete trait definitions for =PollClientBinding=,
=PollServerBinding=, =PollBindingCleanup=, all associated states, and all typed
slots. A small external =no_std + alloc= fake binding must compile using only
public documented APIs before GATE-1 closes.

** AR3-07: Cleanup executor loss has two incompatible terminal models

The exact residual =CleanupTask=/=CleanupExecutor= contract in
=docs/design.md= and ADR-0006 says that an executor which accepts and then drops
a task commits the task's pre-reserved durable
=ResidualExternalState= fallback. The exact =cleanup-task= state projection
instead transitions =ExecutorOwned:executor_drop= and
=ExecutorPollClaimed:executor_drop= to =ManualOwned= by recovering the complete
task envelope.

After =CleanupExecutor::try_spawn= returns =Ok(())=, the arbitrary executor owns
the task and exposes no return channel to Servient. Recovery to a Servient
manual slot and terminal residual commitment are different observable outcomes;
both cannot be the exact transition. The current design also needs to explain
how its destructor safeguard is reconciled with ADR-0011's rule that a
destructor is never the only cleanup mechanism.

Required closure: select one implementable accepted-executor failure contract,
project it identically in the public task API and state machine, and prove that
the complete work object or durable residual fact cannot disappear.

** AR3-08: Most detailed domains still depend on the v4.8 monolith

Only planning and binding SPI currently have registered v4.9 domain
specifications. =docs/spec/README.md= still lists foundation, documents,
interaction core, security/codecs, subscriptions/emissions, Servient,
Discovery client, and profiles/verification as planned owners, and explicitly
states that unmigrated monolith content is not implementation-ready.

This is not merely editorial. The unresolved =CleanupOutcome=,
=CleanupTask=/executor, subscription facade, produced-handle lifecycle, status,
and shutdown contracts used by the new binding specification still derive from
=docs/design.md=. Their exact owner cannot be validated independently until the
corresponding domain specification exists and the residual monolith clauses are
retired or indexed narrowly.

Required closure: finish the single-owner specifications required by the first
implementation tranches, reconcile their API/state projections, and mark each
superseded monolith section as historical rather than simultaneously active.

** AR3-09: Gate and work-package projections are structurally present but not executable

The WP DAG now gives a sensible package order and records cross-crate migration,
old API removal, evidence keys, and workloads. Its structural improvement
resolves the v4.8 “requirements index only” problem. It still cannot authorize
implementation while WP-200/WP-300/WP-400 depend on the undecided public
surfaces above.

In addition, the GATE-1, GATE-2, and GATE-6 rows in
=docs/refactor-gates.csv= still name the rejected
=architecture-review-02-v4.8-rejected= evidence and do not list the active
planning/binding specifications or ADR-0010/ADR-0011 among their artifacts.
The current checker-name mismatch demonstrates that structural checks are not
yet a stable closure pipeline.

Required closure: reconcile the contracts first, then update gate artifacts,
checks, review evidence, and work-package inputs to the same v4.9 revision. A
green DAG parser alone does not make an ambiguous package implementable.

* Refactor admission decision

Do not resume the coordinated runtime refactor yet. The shortest safe closure
sequence is:

1. reconcile ADR-0003/ADR-0006 with ADR-0010/ADR-0011 and freeze one cleanup and
   cancellation vocabulary;
2. freeze complete host/static registration, compiler/artifact, activation-gate,
   and all constrained trait/slot signatures;
3. make route cancellation and subscription/task handoff type-correct against
   the exact state transitions;
4. migrate the Servient/subscription/cleanup detailed owners out of the
   monolith for the first affected work packages;
5. update gate artifacts, checker expectations, and work-package inputs; and
6. rerun the complete artifact suite plus external host and constrained fake-
   binding compile fixtures before changing runtime code.

The architecture now describes where the engine should converge. It does not
yet remove the remaining choices about how ownership terminates, how an
activation gate opens, or how a third-party constrained binding implements the
frozen surface. Those choices must be made once in the design rather than
during parallel crate refactors.
