#+title: Architecture Review 02: Architecture Backbone and Refactor Admission
#+date: 2026-07-19
#+status: Blocking
#+reviewed_design: v4.8 detailed-design candidate
#+next_design: v4.9 architecture-closure candidate

* Executive verdict

The v4.8 candidate is not implementation-ready. It contains substantial useful
detail, but the architecture backbone is not visible enough to constrain those
details. Compiled-plan ownership, Protocol Binding compiler/runtime integration,
server-route driving, long-lived binding memory, and deployment remain choices
that an implementer would have to invent. Several already frozen API and state
artifacts directly contradict their own cleanup and ownership prose.

Runtime migration remains paused. The next target is v4.9, beginning with the
modular architecture backbone under =docs/architecture/= and ADR-0007 through
ADR-0009. The v4.8 monolith is a migration input, not a design that can be made
implementation-ready by appending more API fragments.

GATE-3 remains a valid carry-forward decision for the engine-side Directory
client boundary. GATE-1, GATE-2, GATE-4, GATE-5, and GATE-6 remain open.

* Scope and method

This review evaluated:

- the updated concerns in =docs/thinking/2026-07.org=;
- =PLAN.md= and the v4.8 =docs/design.md=;
- ADR-0001 through ADR-0006, amendments, API ownership, state, resource,
  performance, requirement, governance, and work-package artifacts;
- the current foundation, TD, core, shared binding utility, Servient,
  Discovery, umbrella, and Zenoh source/manifests; and
- adversarial ownership, cancellation, terminal-state, resource, wakeup, and
  deployment traces.

The review distinguishes architecture decisions from exact domain-contract work.
The new backbone freezes the former. It does not claim that the existing v4.8
API/state/resource artifacts already implement the latter.

* Accepted architecture backbone

The v4.9 architecture proceeds from core to edge:

1. W3C TD/TM document fidelity and protocol-neutral semantic values;
2. immutable logical plans and bounded binding artifacts;
3. Protocol Binding compiler and execution SPI;
4. Servient-owned plan/lifecycle/scheduling/cleanup transactions;
5. application composition and deployment; and
6. exact API, state, resource, performance, and migration projections.

The target modules are:

| Module | Owns | Must not own |
|-
| Foundation | Resource/work/time/generation primitives | TD, plans, runtime, protocol behavior |
| TD/TM | Lossless models, builders, Serde, validation, W3C defaults | Selection, runtime cache, transport |
| Core | Protocol-neutral plans, interactions, handlers, security/codec contracts, binding SPI semantics | Compiler algorithms, handles, global schedulers, concrete queues, protocol I/O |
| Planning | Effective-form/security/target resolution, capability index, compiler-extension coordination | Binding execution, Servient state, protocol I/O |
| Servient | Registration snapshot, plan-set lifetime, application handles, lifecycle, dispatch, scheduling, cleanup/status | Protocol syntax/I/O, TD reparsing, implicit Directory service |
| Discovery | Engine-side client sessions/processes/watches/source envelopes | Directory service, storage, server query/redaction |
| Concrete binding | Protocol compiler artifact, I/O, correlation, native flow control/multiplexing, auth extraction, cleanup | Handlers, Servient registry, shared defaults, cross-binding scheduling |
| Umbrella | Composition and re-exports | Runtime implementation |

ADR-0008 renames the target shared compiler package to
=clinkz-wot-planning=. The existing =protocol-bindings/core= package is a WP-200
migration source.

* Architecture blockers found in v4.8

** AR2-01: Normative authority is not single-owner (critical)

=docs/design.md=, amendments, ADRs, work packages, and machine artifacts repeat
observable behavior. The checker assumes all requirement definitions occur in
one design file, preventing safe decomposition. ADR-0002 still contained the
source/signal cancellation model that ADR-0006 removed.

Resolution started: ADR-0007 defines the hierarchy, PLAN is status-only, and
ADR-0002 now explicitly uses ADR-0006's owned call. Remaining v4.8 material must
move into registered domain specifications before gate closure.

** AR2-02: Compiled plans have no complete integration or lifecycle (critical)

The candidate names =LogicalInteractionPlan=, =BindingPlanRef=,
=PlanCompiler=, build input/output, and capability indexes, but does not freeze:

- the complete plan-set owner and publication boundary;
- a protocol binding compiler artifact and its install/release path;
- exact eager versus lazy states and first-use failure;
- generation-scoped pin, drain, and reclamation;
- the atomic association between compiler and execution registration; or
- how an =OutboundRequest= references the selected binding artifact.

ADR-0008 and the plan-lifecycle backbone close the architecture choice. The
planning and binding SPI specifications, state machines, limits, and workloads
still require exact projections.

** AR2-03: Protocol Binding integration/deployment was undefined (critical)

The implementation uses Cargo dependencies/features and explicit
=ServientBuilder= trait-object registration. The only build script generates
foundation resource code. There is no =dylib=/=cdylib= plugin ABI, loader,
automatic inventory, or safe unload protocol.

ADR-0009 freezes ordinary downstream Rust crates, complete explicit startup
registration, and process/container/firmware rollout. =build.rs= discovery,
direct Rust trait-object dynamic loading, runtime registration replacement, and
hot unload are outside v1.

** AR2-04: Server route lifecycle is not implementable (critical)

The exact-looking host surface has multiple ownership holes:

- =activate= consumes a prepared guard but ordinary =CoreResult= cannot return
  it on failure;
- cleanup borrows a guard while =PendingCleanup= claims to transfer full
  ownership;
- guard wrappers and registration/readiness/context types have no complete
  public constructors/accessors or type-erasure contract;
- readiness is a bare function fragment, not an exact trait;
- preparation visibility and endpoint reservation classification are prose
  without registration fields;
- registration-wide =poll_accept= has no route/guard identity, unique waker
  owner, or scoped terminal event; and
- self-driving behavior depends on undefined =ctx.dispatch= and incomplete
  status delivery.

ADR-0009 selects engine-orchestrated, route-scoped progress and removes the
self-driving path from v1. An exact server-binding specification and ADR for
route/guard lifecycle are still required before GATE-1 or GATE-2 can close.

** AR2-05: Host binding cancellation transfers status but not work (critical)

The owned-call direction is correct, but its current exact schema is not yet
closed:

- one =CleanupTransferContext= is reused across start cancellation and active
  subscription stop despite distinct operations and deadline linearization;
- a transparent handle cannot materialize the required bounded subject/owner
  identity by itself;
- =PendingCleanup= does not specify how the complete call object reaches the
  next poll owner;
- executor/task wake-at-deadline and budget semantics are incomplete;
- drain-expiry, footprint overflow, task rejection/drop, and cleanup-handle
  acknowledgement are missing from the state projection; and
- retry classification is duplicated without an equality rule.

The binding SPI specification must split reusable cleanup reservation from a
phase-specific transfer context and must move the complete object, not only a
=CleanupRecord=.

** AR2-06: Constrained binding slots are opaque to binding implementers (critical)

Core-owned concrete request/subscription/response/emission slots have private
state but no generic/associated storage contract through which an external
protocol crate can install its cursor. A binding-owned side table would recreate
the unbounded and generation-unsafe state the design forbids.

The v4.9 SPI must use associated state with caller-owned typed slots or an
equally explicit bounded arena. Size, alignment, generation reuse, state drop,
and cleanup must be testable. Incremental prepare and form contribution also
need =WorkBudget= and retained cursors instead of whole-Thing unbudgeted calls.

** AR2-07: Long-lived binding memory and ingress are not admitted (critical)

The design bounds initial host call objects but not:

- lifetime growth of a call after its first poll;
- prepared/active route guards and readiness tokens;
- installed host subscription drivers;
- protocol response/cancellation buffers; or
- pre-dispatch external-input queues behind route acceptance.

Every long-lived object needs an immutable lifetime upper bound plus actual
verification. Ingress needs explicit per-route/per-binding/global item and byte
limits. Poll-time transient allocations and shared payload leases require
separate ledger rules.

** AR2-08: Response and emission start ownership is lossy (critical)

Constrained =start_response= consumes an owned response even though a
pre-acceptance backpressure error promises to leave the response opportunity
with the caller. The signature cannot satisfy that guarantee. Emission start
has the same input-retention question. Host response progress is synchronous in
one surface and pending/cancellable in another.

The binding SPI must return a typed rejected input or use an explicit owned
input slot and align host/constrained terminal semantics.

** AR2-09: Subscription terminal and cleanup phases are conflated (high)

Remote terminal delivery may return pending/residual cleanup without receiving
an engine-issued phase context. Start cancellation, remote terminal, explicit
stop, implicit drop, and driver cleanup require separate first-cause and
deadline ownership. =start_stop= also lacks a =Context= and can lose a wake
between start and first poll.

The exact subscription specification and state projection must separate process
terminal classification from resource cleanup while preserving one public
terminal event.

** AR2-10: State machines omit legal terminal dispositions (critical)

=binding-route= and =expose= project cleanup as pending-only in transitions even
though APIs allow synchronous complete and direct residual outcomes. Activation
failure retention, lifecycle callback races, drain-deadline residual, and
cleanup-task handoff are absent. Some subscription failure transitions also
confused process failure with residual resource state.

State artifacts must use outcome-specific transitions and semantic negative
tests rather than checking only that named fragments exist.

** AR2-11: Fair progress and wake ownership are incomplete (high)

Readiness can be awaited serially, allowing one never-ready route to block all
others. Several start/poll pairs have no register-then-recheck rule. Whole-Thing
contribution and constrained preparation can process thousands of plans without
a budget or cursor.

The runtime and SPI specifications must freeze fair retained cursors, one
overall deadline, per-owner work quanta, wake registration, and zero-budget
behavior.

** AR2-12: Performance evidence does not cover the new ownership model (critical)

Missing workloads include:

- cold/hot compiled-plan lifecycle and reclamation;
- third-party binding integration and startup;
- route/guard/readiness footprints and maximum ingress;
- lifecycle callback cancellation at every phase;
- owned-call late results, drain expiry, task/manual cleanup transfer;
- constrained slot cancellation and rejected-input preservation; and
- slow/never-ready route fairness.

Performance manifests require deterministic virtual time and exact fixture
identity before GATE-5 can close.

* Current implementation deviations

The code confirms rather than resolves the review:

- core still owns =EventBroker=, publisher sinks, queue-backed subscriptions,
  and merge behavior in =core/src/event.rs=;
- current core exposes old =BindingRequest= and async/self-driving
  =ServerBinding= contracts;
- Servient registers bare =Arc<dyn ServerBinding>= and
  =Arc<dyn ClientBinding>= and has a normal optional concrete Zenoh dependency;
- Zenoh server scans TD data, owns Tokio drain tasks, queues ingress, and uses
  blocking publication/cleanup paths;
- Zenoh client recompiles/caches form data internally; and
- Discovery still exports in-memory service/storage behavior.

These are migration sources. They do not authorize weakening the v4.9
architecture.

2. Turn =docs/design.md= into a revision/source manifest and move detailed
   behavior into single-owner domain specifications.
3. Add the server route/guard and cleanup/cancellation decisions required by
   the findings above.
4. Update API, state, resource, performance, requirement, governance, and
   work-package artifacts to v4.9.

   cannot be converted into passing evidence because it reviewed and rejected
   v4.8.
7. Resume foundation and runtime implementation only after all required gates
   are closed.
